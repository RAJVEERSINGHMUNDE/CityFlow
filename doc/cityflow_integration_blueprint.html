Pillar 1 Multilingual NLP + Hawkes Process ▶

Hawkes Process: do not implement. A Hawkes process requires dense, precisely timestamped event streams. Your dataset spans 5 months with only 8 173 events (~55/day city-wide). The inter-arrival times are too coarse and sparse for the self-excitation parameters α/β to converge meaningfully. The math is right but the data volume is wrong.
NLP weak-label classifier: build this. You have 840 "slow", 385 "no problem", 448 "moving", 24 "closed" labels inside descriptions. That is enough for a lightweight binary classifier: ImpactSignal ∈ {Contained, Disrupted}. Use sentence-transformers/LaBSE (multilingual, handles Kannada, runs on CPU, ~90 MB) rather than IndicBERT. Fine-tune a logistic head on your labelled descriptions.
What to actually build
# src/simulator/nlp_impact.py (NEW FILE) # ~80 lines, no GPU needed from sentence_transformers import SentenceTransformer from sklearn.linear_model import LogisticRegression import numpy as np, re, joblib, os WEAK_LABELS = { 'no problem': 0, 'normal': 0, 'moving': 0, 'slow': 1, 'closed': 1, 'blocked': 1, 'gridlock': 1, 'ನಿಧಾನ': 1, # "slow" in Kannada 'ಸಮಸ್ಯ': 1, # "problem" 'ನಿಂತಿದೆ': 1, # "standing/stopped" } class NLPImpactClassifier: MODEL_ID = "sentence-transformers/LaBSE" def fit(self, df): desc = df['description'].dropna() # Weak labelling from keyword presence labels, texts = [], [] for text in desc: for kw, lbl in WEAK_LABELS.items(): if kw in text.lower(): texts.append(text); labels.append(lbl); break self._encoder = SentenceTransformer(self.MODEL_ID) X = self._encoder.encode(texts, batch_size=64, show_progress_bar=True) self._clf = LogisticRegression(max_iter=500).fit(X, labels) joblib.dump((self._encoder, self._clf), 'nlp_model.pkl') def predict_impact(self, text: str) -> dict: X = self._encoder.encode([text]) prob = self._clf.predict_proba(X)[0] return {'disrupted_prob': round(float(prob[1]), 3), 'label': 'Disrupted' if prob[1] > 0.5 else 'Contained'}

Wire into severity_model.py: after the GBM predicts resolution time, call nlp_impact.predict_impact(event_dict.get('description','')) and add nlp_disruption_prob to the severity result dict. Expose it as a new field in GET /api/severity/<id>. This directly upgrades the severity score with ground-truth radio signal rather than purely structural features.

Corridor-weight modifier: replace the Hawkes Process cascade idea with a simpler, honest version. After NLP classification, multiply the disrupted_prob by a corridor weight derived from historical closure rates per corridor (already computed in hotspot_analyzer). This is the ΔC formula — corridor-conditioned capacity drop — without requiring the Hawkes temporal density you don't have data for.
Pillar 2 Geometric Footprints — Polygon-Constrained Routing ▶

Full polygon masking: do not implement. Only 86 of 8 173 rows (1%) have non-empty route_path coordinates. The area-intersection formula C_effective = C_base × (1 − intersection/area) is mathematically sound but will fire on fewer than 1% of events. Building a full geospatial masking pipeline for 86 rows is a high cost / low return engineering choice.
Edge-snapping for linestrings: build this. For the 86 events that do have coordinates, use the linestring to snap directly to OSM graph edges — this replaces the current single-point epicenter with a multi-node closure zone. This is concrete, correct, and takes ~40 lines added to graph_engine.py.
What to actually build — linestring edge masking
# Add to src/simulator/graph_engine.py import ast, osmnx as ox def mask_edges_from_linestring(G, route_path_json: str) -> set: """ Given a JSON linestring from route_path column, snap each coordinate pair to the nearest OSM edge and return the set of (u,v) edges to close. Falls back to empty set if route_path is empty/invalid. """ try: coords = ast.literal_eval(route_path_json) if not coords or coords == []: return set() closed = set() for lat, lon in coords[::3]: # sample every 3rd point — dense enough u = ox.distance.nearest_nodes(G, X=lon, Y=lat) # Mark all edges incident to this node as closure candidates for _, v in G.out_edges(u): closed.add((u, v)) return closed except Exception: return set()

In app.py, when an event has a non-empty route_path, pass the result of mask_edges_from_linestring() as the initial closed_edges seed to simulate_congestion_shockwave() instead of computing the closure radius from a single point. For the 99% of events without linestrings, behaviour is unchanged.

Demo value: This directly fixes the eval log issue where "barricades fractured the graph." A linestring closure is topologically consistent — the edges being closed are exactly the road the incident lies on, not a Euclidean circle that might bleed into a flyover above it.

What to actually build
# src/simulator/survival_model.py (NEW FILE ~70 lines) import pandas as pd, numpy as np from lifelines import CoxPHFitter import warnings; warnings.filterwarnings('ignore') class ClearanceForecaster: """ Cox Proportional Hazards model for incident clearance time. Outputs a survival curve S(t|X) and a confidence interval for the 80th percentile clearance time. """ def fit(self, df: pd.DataFrame): df = df.copy() df['start_dt'] = pd.to_datetime(df['start_datetime'], utc=True, errors='coerce') df['closed_dt'] = pd.to_datetime(df['closed_datetime'], utc=True, errors='coerce') df['T'] = (df['closed_dt']-df['start_dt']).dt.total_seconds()/60 df['E'] = 1 # all observed (no censoring in this dataset) # Keep usable rows df = df.dropna(subset=['T','veh_type','corridor']) df = df[(df['T'] > 2) & (df['T'] < 43200)] # Dummy-encode categoricals df['is_bmtc'] = (df['veh_type']=='bmtc_bus').astype(int) df['is_heavy'] = (df['veh_type']=='heavy_vehicle').astype(int) df['is_hcorridor'] = df['corridor'].apply( lambda c: 0 if c=='Non-corridor' else 1) df['requires_closure'] = df['requires_road_closure'].fillna(False).astype(int) features = ['T','E','is_bmtc','is_heavy','is_hcorridor','requires_closure'] self._cph = CoxPHFitter(penalizer=0.1) self._cph.fit(df[features], duration_col='T', event_col='E') self._is_fitted = True print(self._cph.summary[['coef','exp(coef)','p']].to_string()) def predict_clearance(self, event_dict: dict) -> dict: row = pd.DataFrame([{ 'is_bmtc': int(event_dict.get('veh_type','')=='bmtc_bus'), 'is_heavy': int(event_dict.get('veh_type','')=='heavy_vehicle'), 'is_hcorridor': int(event_dict.get('corridor','Non-corridor')!='Non-corridor'), 'requires_closure':int(bool(event_dict.get('requires_closure',False))), }]) sf = self._cph.predict_survival_function(row) t_vals = sf.index.values # Find t where S(t) first drops below 0.5 (median) and 0.2 (80th pct) s_vals = sf.iloc[:,0].values t50 = float(t_vals[np.searchsorted(-s_vals, -0.5, side='right')]) t80 = float(t_vals[np.searchsorted(-s_vals, -0.2, side='right')]) return { 'median_clearance_min': round(t50, 0), 't80_clearance_min': round(t80, 0), 'survival_at_30min': round(float(sf.iloc[:,0][sf.index<=30].iloc[-1]), 3), 'survival_at_60min': round(float(sf.iloc[:,0][sf.index<=60].iloc[-1]), 3), }

Train at startup alongside the severity model in app.py. Merge the output into the /api/severity/<id> response under a new key clearance_forecast. The dashboard can then show: "80% probability this clears within 47 min — diversion active for next 45 min."

This also fixes the time-of-day routing invariance bug you found. The current system applies a uniform 1.5× multiplier to all edges, which doesn't change Dijkstra's path choice. Replace the point-estimate resolution time in manpower.py with the Cox t80 output — the shift_duration_hours will now vary correctly by vehicle type and corridor, making the manpower allocation data-driven rather than rule-based.
Pillar 4 Contrastive Learning — Weakly Supervised Embeddings ▶
~1 200labellable description pairs (rough)
Triplet lossneeds 3 × N samples
Full contrastive learning with triplet loss: do not implement. Triplet loss requires careful online negative mining and typically needs tens of thousands of labelled pairs to converge. With ~1 200 labellable descriptions and no structured outcome ground truth beyond the description text itself, the anchor/positive/negative construction is circular — you are labelling descriptions with descriptions. The math is impressive but the data doesn't support it.
What to build instead — outcome-conditioned feedback. The /api/feedback endpoint already exists in your codebase with observed_severity, actual_officers, diversion_effective fields. Feed those back into the NLP classifier (Pillar 1) as confirmed labels. After 50–100 real feedback entries, the LaBSE logistic head can be re-fitted in under a second. This is the "weakly supervised learning loop" materialised as an actual running system — simpler than triplet loss, but genuinely post-event learning.
Honest architecture for the learning loop
# Add to src/api/storage.py — new function def get_confirmed_labels() -> list[dict]: """Return feedback rows that can serve as confirmed NLP training labels.""" con = _connect() rows = con.execute(""" SELECT f.event_id, f.observed_severity, f.diversion_effective, e.description FROM feedback f JOIN events e ON f.event_id = e.id WHERE e.description IS NOT NULL """).fetchall() con.close() return [dict(r) for r in rows] # In nlp_impact.py — add incremental re-fit def retrain_from_feedback(feedback_rows: list[dict]): """ Re-fit the logistic head using confirmed operator labels. Runs in <1s; called from POST /api/feedback after N=50 new entries. """ texts = [r['description'] for r in feedback_rows] labels = [0 if r['observed_severity']=='Green' else 1 for r in feedback_rows] X = _encoder.encode(texts, batch_size=64) _clf.fit(X, labels) # warm-start refit

This closes the "post-event learning" gap from the problem statement with something that actually runs, rather than a theoretical triplet loss training pipeline that would take days and GPU time.
